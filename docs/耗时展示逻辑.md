# 耗时展示逻辑说明

## 一、耗时显示在哪里

**最终位置**：机器人「已保存～」回复消息的**第 4 行**。

**调用链**：
```
message_handler.py
  _success_reply() [行 50-65]
    lines = [已保存, text, 总耗时]
    if tlist:
        lines.append("📊 " + " ".join(parts))   # 分步耗时
    reply_message(chat_id, msg_id, "\n".join(lines), ...)
       ↓
feishu/bot.py
  reply_message() [行 113-145]
    payload = {"msg_type": "text", "content": json.dumps({"text": text})}
    POST /im/v1/messages/{msg_id}/reply
```

**预期展示格式**：
```
已保存～ 📚
已记录：xxx...
⏱ 总耗时 81.7s
📊 抓取8.2s AI总结15.3s 去重0.5s 写表2s 建索引45s
```

## 二、分步耗时从哪里来

### 链接流程 `_handle_url_impl`

| 步骤   | short_desc | 追加位置 |
|--------|------------|----------|
| 抓取   | 抓取       | 86       |
| AI总结 | AI总结     | 120（有全文时） |
| 去重   | 去重       | 156      |
| 写表   | 写表       | 180      |
| 建索引 | 建索引     | 185      |

成功时 **187 行**：`_success_reply(..., timing=list(TIMING_BREAKDOWN))`

### 图片流程 `_handle_image_impl`

- **fallback 分支**（无链接、有视频信息）：取图、识图、搜链、抓取、建索引 → **267 行** 传入 timing
- **多链接分支**：取图、识图、抓取（每个 url 只记录一次）→ **303 行** 传入 timing

---

## 三、可能不显示的原因

1. **飞书消息长度限制**：text 字段可能有 4096 字符等限制，消息过长时被截断（我们的分步耗时通常很短，理论上不会被截断）
2. **多实例**：Koyeb 多实例时 `TIMING_BREAKDOWN` 不共享，已通过 `timing=list(TIMING_BREAKDOWN)` 传入快照规避
3. **部署未更新**：线上仍是旧版本代码
4. **tlist 为空**：未走到 append 的分支，或传入的 timing 为空（如异常提前 return）
