# 读取与解析耗时分析与优化建议

## 一、技术方案与典型耗时

### 链接场景 (handle_url)

| 步骤 | 操作 | 典型耗时 | 说明 |
|------|------|----------|------|
| 1. extract_metadata | HTTP 抓取 + trafilatura + BeautifulSoup | **5–15s** | 目标站响应慢（公众号等）时更长 |
| 2. generate_summary | AI 总结（gemini-2.5-pro） | **15–35s** | LLM 调用，可能含 deepseek 兜底 |
| 3. find_record_by_content_url | 飞书 list_records(150) | **2–5s** | 去重查询 |
| 4. add_record | 飞书写入单条 | **1–3s** | |
| 5. add_to_index | ChromaDB + Embedding API | **10–60s** | **首次调用**会重建索引（200 条 × embedding） |

**链接总耗时**：约 **35–120s**（首次更多）

---

### 图片场景 (handle_image)

| 步骤 | 操作 | 典型耗时 | 说明 |
|------|------|----------|------|
| 1. get_image_for_vision | 飞书获取图片 | **1–3s** | |
| 2. analyze_image | 多模态识别链接 | **8–20s** | 1–2 次 vision API（kimi/gemini） |
| 3. find_best_video_url | 联网搜索链接（无 URL 时） | **5–15s** | supermind-agent |
| 4. extract_metadata | 抓取链接 | **5–15s** | 同上 |
| 5. generate_summary | AI 总结 | **15–35s** | 同上 |
| 6. add_record | 飞书写入 | **1–3s** | |
| 7. add_to_index | ChromaDB 建索引 | **10–60s** | 同上 |

**图片总耗时**：约 **45–150s**（首次更多）

## 二、主要瓶颈与优化方向

### 1. add_to_index / ChromaDB 首次冷启动（最大瓶颈）

**现状**：首次调用会 `_rebuild_index()`，拉取 200 条飞书记录并批量做 embedding，可能 **30–60s**。

**可行优化**：
- **异步建索引**：入库后先立即回复用户，`add_to_index` 放到后台/延迟执行，不阻塞主流程
- **降低 limit**：`list_records(limit=200)` → `50`，减少首次重建时间
- **持久化 ChromaDB**：使用持久化存储，避免每次重启都重建
- **按需建索引**：仅对「提问」需要的记录建索引，入库时可选跳过

### 2. generate_summary（第二瓶颈）

**现状**：gemini-2.5-pro 单次约 15–35s，失败时还可能有 deepseek 兜底。

**可行优化**：
- **换更快模型**：如 gemini-2.0-flash、deepseek-v3 等，通常 5–15s
- **缩短输入**：只取前 2000–3000 字总结，减少 token 与延迟
- **可配置跳过**：支持「仅记录不总结」，用 trafilatura 的前 500 字作摘要

### 3. extract_metadata（第三瓶颈）

**现状**：目标站响应慢时（尤其公众号）可达 10–15s。

**可行优化**：
- **缩短超时**：`timeout=15` → `10`，快速失败
- **并行/缓存**：多处需要同一 URL 时，可考虑缓存或并行请求（图片多链接场景）
- **限流/队列**：对易反爬站点做限速或队列，避免反复超时

### 4. analyze_image（图片场景主要成本）

**现状**：1–2 次多模态调用，约 8–20s。

**可行优化**：
- **单次调用**：合并「找链接」和「提取视频信息」为一次 prompt
- **更轻模型**：在效果可接受前提下，尝试更快的 vision 模型

---

## 三、建议实施顺序

1. **异步 add_to_index**（改动小、收益大）：入库后先回复，索引后台执行
2. **换用更快摘要模型**：如 gemini-2.0-flash
3. **缩短全文**：总结时只取前 2500 字
4. **降低 _rebuild_index 的 limit**：200 → 50 或 80
